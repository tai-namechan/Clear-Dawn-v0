# AI 支援仕様

対象フェーズ: Phase 4

## 目的

蓄積されたデータ（TOP Matrix・記録・実行履歴・振り返り）をもとに、
優先順位付けと振り返りを支援する。AI は決定しない。決めるのはユーザーである。

## 画面

| 画面 | ルート | 概要 |
|---|---|---|
| AI 支援 | GET /ai | 提案の閲覧・生成・採用 |

形態（独立画面のみか、各画面への埋め込みパネル併用か）は **未決定**（M7 前に確定）。

## 主要ユースケース（v0 スコープ案）

| ユースケース | 入力 | 出力 |
|---|---|---|
| 「今やるべきこと」候補の提案 | マトリクス全体 + 直近の実行履歴・記録 | セル項目候補（採用時にセル項目として追加） |
| 振り返り下書き生成 | 当日 / 当週の完了実績・記録 | 日次・週次振り返りの所感ドラフト |
| 領域バランスの指摘 | 領域別の完了実績・記録 | 偏りの指摘テキスト |

## 設計方針

- 提案は必ず「採用 / 破棄」のユーザー操作を挟む。AI が直接データを書き換えない
- 送信するデータは自分のデータのみ。プロンプトに他ユーザー・秘密情報を含めない
- 提案・生成の履歴（`ai_suggestions`）を保存するかは **未決定**（保存する場合は採用状況も記録）
- 応答は非同期（Queue）前提で設計し、画面はポーリングまたは通知で更新する

## 未決定事項（M7 着手前に確定）

| 項目 | 論点 |
|---|---|
| プロバイダ | どの LLM API を使うか。API キーは環境変数管理（リポジトリに書かない） |
| コスト管理 | 呼び出し回数制限・トークン上限 |
| 形態 | 独立画面のみ vs 各画面埋め込みパネル併用 |
| ログ保存 | 提案履歴を DB に残すか、残す場合の保持期間 |

## データ（ドラフト）

テーブル定義は [../../data/tables.md](../../data/tables.md) の `ai_suggestions`（案）を参照。

## セキュリティ

- API キー・エンドポイント等の秘密情報はコード・ドキュメントに書かない（環境変数のみ）
- 外部 HTTP を伴うため、テストでは `Http::preventStrayRequests()` 相当のガードを必須とする
