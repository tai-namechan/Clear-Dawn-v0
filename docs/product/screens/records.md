# 記録（体重・睡眠・筋力・野球）とグラフ仕様

対象フェーズ: Phase 2.5

## 目的

身体・パフォーマンスの状態を数値で記録し、推移をグラフで確認する。
「将来どうなっていたいか」（TOP Matrix の将来像行）に対する現在地を示す。

## 画面

| 画面 | ルート | 概要 |
|---|---|---|
| パフォーマンス管理ハブ | GET /records | 当日サマリ + 食事/コンディションへの入口。カード内は直近7日のミニ推移 |
| コンディション | GET /records/condition | **今日 / 推移 / 設定** タブ。チェックイン + 基本測定（今日）とチャート（推移）を分離 |
| 記録 | GET /records/{metric} | メトリクス別 + 入力フォーム + 推移グラフ |

`{metric}` は `weight` / `sleep` / `strength` / `baseball` から開始する。

コンディション「今日」は入力専用（大きな空チャートは出さない）。「推移」は分析専用。

### ハブ（GET /records）の構成

1. **上部サマリ** — 選択日の摂取 kcal / PFC / 主要メトリクス（当日スナップショット）
2. **食事記録カード** — 直近7日の kcal 折れ線 + PFC 積み上げ（ミニ）。記録 CTA → `/meals`
3. **コンディション管理カード** — 直近7日の体重・睡眠ミニ推移 + 筋力ミニチャート。記録 CTA → `/records/condition`

- 当日の詳細数値・区分別食事は上部サマリまたは各記録画面で扱う。カード内に朝昼夜区分や当日 PFC ドーナツを再掲しない
- グラフは期間固定（選択日を含む直近7日）。データが無いときは空グラフではなくプレースホルダ文言
- 集計は既存 Query（`GetNutritionChartQuery` / `GetMetricChartQuery` / `GetStrengthChartQuery`）を再利用する

## メトリクス別の内容

| タブ | 記録内容 | 単位・形式 |
|---|---|---|
| 体重 | 日別の体重 | kg（小数 1 桁） |
| 睡眠 | 就寝・起床時刻または睡眠時間 | 時間（分単位） |
| 筋力 | 種目 × セット × 重量 × 回数 | 構造化レコード |
| 野球 | 打撃 / 投球などの成績・練習記録 | 構造化レコード |

- 体重・睡眠は「1 日 1 レコード」の単純数値系
- 筋力・野球は構造を持つため専用設計とする（下記「スキーマ戦略」参照）

## 主要操作

| 操作 | 入力 | 挙動 |
|---|---|---|
| 記録入力 | 日付 + 数値（または構造化フォーム） | upsert（単純数値系は unique(user_id, metric, date)） |
| 期間指定グラフ | 期間（週 / 月 / 3 ヶ月 / 年） | 集計クエリで取得し折れ線等で表示 |
| 過去分の修正 | 日付指定 | 上書き更新 |

## グラフ方針

- チャートライブラリは **ECharts**（`echarts`）。配色はデザイントークン `--chart-1`〜`--chart-5` を使用（[design-system.md](../../design/design-system.md)）
- グラフ描画時に全件取得しない。期間指定 + 必要なら日次・週次集計で件数を抑える
- **残作業**: 週次平均などの集計クエリ、筋力チャート UI 配線、仕様タブ（strength / baseball 構造化）とのギャップ解消。詳細は [progress.md](../../progress.md)

## スキーマ戦略（ハイブリッド推奨・未確定）

| 対象 | 案 | 内容 |
|---|---|---|
| 単純数値系（体重・睡眠） | 汎用テーブル | metric 定義 + `metric_records`（value, recorded_on） |
| 構造化系（筋力・野球） | 専用テーブル | 筋力は routine 実行ログとの関係も含め M4 設計時に確定 |

M4 着手前に ADR として確定する。詳細ドラフトは [../../data/tables.md](../../data/tables.md)。

## TOP Matrix との接続

- メトリクスは `life_area_id` を任意で持ち、「将来どうなっていたいか」の目標と
  領域経由で突き合わせられるようにする
- 週次振り返り画面に週間サマリを参照表示する（[reviews.md](./reviews.md)）

## 設計方針

- 日次データは `date` 型カラムで持つ（datetime にしない）。**日付の正はユーザー TZ**（`users.timezone` → `UserTimezoneResolver`。未設定時は `Asia/Tokyo`）
- 想定件数: メトリクスあたり 365 件/年 × 複数メトリクス。グラフは期間指定で取得
- 記録の削除は物理削除でよい（履歴性より入力ミス訂正を優先）

## 認可

- 自分の記録のみ操作可能（Policy）
