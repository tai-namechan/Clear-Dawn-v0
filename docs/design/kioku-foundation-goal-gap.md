## 2. ゴール定義（decision として保存可能な形）

今日の議論は、既存 `memory_type: decision` の `structured_data` に **不足なく収まる**（スキーマ追加不要）。これはデータモデル妥当性の証拠。

| フィールド | 内容（要約） |
|---|---|
| title | 記憶基盤キオクのゴールは「海馬の外付け」 |
| situation | 海馬に勝てない感覚 → プロダクトの存在意義への迷い |
| constraints | 連想/文脈/直感では勝てない・個人開発・堀は10年分の自分データ |
| options | A 海馬再現（却下）/ B 苦手だけ外付け（採用） |
| decision | 劣化コピーではなく外付け HDD。苦手5つのみ肩代わり |
| reason | 同じ土俵で「より上手く」やろうとした瞬間だけ負ける（電卓と暗算） |
| review_condition | 記憶100件超で週次パターン発見を試作した時、「自動発火」が価値を生むかで再評価 |
| tags / importance | 記憶基盤, ゴール定義, 海馬, 設計思想 / 5 |

> 実装メモ: `DecisionType` の required は `situation` / `decision` / `reason`。`constraints` / `options` / `review_condition` は任意だが seed-k では `review_condition` を必須方針としている。手動保存 or enrich 後の確認で揃える。

## 3. 海馬の苦手5つ ↔ キオクの対応

| 苦手 | 意味 | 現状 | 設計予定 | やりたいこととの差 |
|---|---|---|---|---|
| **正確さ** | 書き換わらない原文 | `raw_content` 保存。reenrich は派生（type/summary/structured）のみクリア | 設計書の raw 不変原則 | ✅ ほぼ合致。UI/API で raw 更新を明示禁止するガードは薄い（運用・コード規約頼み） |
| **持続** | 10年残る | `memories` 永続化 | アーカイブ / Hot-Warm-Cold（監査ドキュメント） | △ 永続はある。10年運用（退避・再enrich・ストレージ階層）は未着手 |
| **自動発火** | 意志なしで思い出す | ❌ なし（ユーザーが検索 / ヨユウが文脈テキストで引くだけ） | Recall 改善は **受動の高速化**。能動プッシュは明示的に薄い | ⚠️ **最大ギャップ**。存在理由そのもの |
| **感情フィルタ** | 必要な時だけ感情を出す | `emotion` type・`sensitive`（Recall 除外） | 感度 0–4（ChatGPT 取込 Phase） | △ 型はある。文脈に応じた出し分けは未設計 |
| **領域またぎ** | 仕事×野球など横断相関 | `memory_links` はタグ/タイトル語の関連（最新100件上限） | 週次パターン / insights（Phase B 付近） | ⚠️ **第二のギャップ**。リンク≠領域横断の発見 |

付帯:

| 概念 | 現状 | 設計 | 差 |
|---|---|---|---|
| リプレイ（固定化） | `captured → enriching → ready` 二段 | 同左 | ✅ |
| 受動 Recall | ヨユウ Today/Chat に最大5件注入 | PR-E で SQL 前処理・cache・計測 | 品質・性能の改善は予定あり。能動化は別物 |
| ゴール記憶自体 | decision として保存可能 | スキーマ確定済み | ✅ 保存オペレーションは人間側 |

## 5. 差分の整理（何がズレているか）

### 5.1 現状 vs 設計予定

設計どおり **「まず保存と検索（＋整理）」から入る** 順序は守れている。

残っている設計上の近場:

1. **PR-E** — Recall の SQL 前処理・短期キャッシュ・計測（受動経路の健全化）
2. **PR-G** — ChatGPT 取込 Phase A（embedding なし）→ Phase B（vector / insights）
3. 監査 F-3/F-4 — LIKE 線形劣化・表示ごとの走査
4. 一覧の cursor 化・一覧 Resource から raw 除外（監査推奨）

これらは「壊れない基盤」であって、**海馬の外付けの魔法**ではない。

### 5.2 設計予定 vs やりたいこと

| やりたいこと | 設計での扱い | ズレ |
|---|---|---|
| 自動発火 | 受動 Recall 改善が先。能動は後回し／曖昧 | 存在理由がロードマップの後ろに沈んでいる |
| 領域またぎ | 100件超・週次バッチ等の将来条件 | 検証が数ヶ月先になり得る |
| コンシェルジュ（週1手紙） | 設計書に無い | **設計に載せるべき新実験** |
| raw 不変の徹底 | 原則あり、実装ガードは緩い | 原則をコード不変条件へ落とし込む余地 |
| 習慣＝堀 | 監査で継続率に触れる程度 | プロダクトKPIとして未定義 |
| MCP 提供側 | 無し | アーキ提案。優先度は低い |

### 5.3 現状 vs やりたいこと（一番痛い差）

```text
現状の価値提供:  「覚えさせる・探す・秘書の文脈に載せる」
やりたい価値提供:「忘れていたことを、システム側から差し込む」
```

前者は実装済み。後者はゼロ。今日の検証が炙り出した最大ギャップ。

## 7. 着手トリガーの具体化案（§16 の置き換え候補）

リポジトリ外設計の「記憶100件超で週次バッチ」は、**存在理由の検証には遅すぎる**。

| 案 | 内容 | 必要な実装 | わかること | 推奨 |
|---|---|---|---|---|
| **A. コンシェルジュ実験（推奨）** | 週1回、手元の記憶をAIに読ませ「忘れている気がする」手紙を1通生成 | プロンプト＋手動 or 超薄い Artisan。バッチ基盤ほぼ不要 | 自動発火に価値があるか（今週〜） | ✅ 今決める実験 |
| B. 100件超・週次バッチ | 設計書どおり本番寄りのパターン発見 | Job・集計・UI・評価 | 領域またぎの本格価値 | 件数が貯まってから |
| C. PR-E のみ先 | Recall 高速化 | cache/SQL | 性能。魔法の有無はわからない | 基盤保守としては必要だが検証の主役にしない |

**推奨の決め方:**

1. **今週**: 案Aで自動発火の価値を検証（成功条件を1行で書く）
2. **並行〜直後**: PR-E で受動経路を壊さない
3. **100件 or 案Aが価値あり**: 領域またぎバッチ（案B）を正式に roadmap へ昇格

案Aの成功条件例（仮）:

> 生成された1通のうち、ユーザーが「忘れていたが今必要だった」と判定した項目が1つ以上ある。

## 8. 境界の明文化（やりたいことを壊さないための線）

| 層 | 方針 | 今守れているか |
|---|---|---|
| **記憶ストア（中核）** | raw_content 不変、退屈な技術、user_id スコープ、再 enrich 可能 | △ ほぼ。不変をコードで強制しきれていない |
| **知能（周辺）** | classify/extract/Recall/コンシェルジュは交換可能。モデル世代で全再処理できる前提 | ✅ 思想は合致 |
| **能動想起** | 「検索UIの延長」にしない。別ユースケース（手紙・通知・ヨユウ差し込み） | ❌ 未着手 |
| **やらない土俵** | 連想の速さ・無意識の整理・直感の再現で海馬と競争しない | ✅ ゴール定義で確定 |
