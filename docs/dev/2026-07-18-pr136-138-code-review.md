## High

### REV-01: チェックイン保存で note / region_tension がデータ消失（確定）

- 対象: `app/Services/UpsertDailyCheckinService.php` / `resources/js/components/routine/DailyCheckinPanel.vue` / `resources/js/pages/Records/Condition.vue:471` / `resources/js/pages/Today/Index.vue:159` / `resources/js/types/todayOps.ts`（`CheckinFormState`）
- 問題: サービスは `updateOrCreate` の値配列を `$attributes['note'] ?? null` 形式で組むため、**リクエストに含まれないキーは null で上書き**される。PR #138 の新 `DailyCheckinPanel` はスライダー 6 項目（sleep_quality / fatigue / muscle_soreness / stress / mood / readiness_self）しか送らないので、Today・Condition どちらから保存しても既存の `note`・`region_tension` が毎回消える。
- 再現: ① note と region_tension 入りのチェックインを保存 → ② 同日に DailyCheckinPanel でスライダーを 1 つ動かして保存 → ③ note / region_tension が null。
- 修正案（推奨）: サービスを「送信されたキーのみ更新」に変更する。例: 更新可能フィールドのホワイトリストに対し `array_intersect_key($attributes, array_flip($fields))` で値配列を作り、`firstOrNew` → `fill` → `save`。コントローラ側は `$request->validated()` のままでよい（`sometimes` ルールなので未送信キーは含まれない）。
- 追加テスト: 既存 checkin（note あり）に対し部分フィールドのみ POST → note が保持されることを Feature テストで担保。

### REV-02: 前日食事コピーの重複ガードなし（確定）

- 対象: `app/Services/CopyPreviousDayMealsService.php:35-50` / `app/Http/Controllers/MealEntryController.php`（copyPreviousDay） / `resources/js/pages/Meals/Index.vue:337,668`
- 問題: 対象日の既存エントリを確認せず無条件 `create` ループ。二重 POST、または既に記録がある日への実行で食事が丸ごと重複する。ガードはクライアントの `saving` フラグのみでサーバ側防御ゼロ。
- 修正案: 仕様判断が必要（→ S-04）。最小修正は「対象日に既存エントリがある場合は 409 or `copied: 0` + メッセージで拒否」。トランザクション内で対象日を `lockForUpdate` して existence チェック。
- 追加テスト: ①二重呼び出しで件数が増えない ②対象日に既存記録ありで拒否 ③空前日で `copied: 0` ④PFC・meal_type・note がコピー元と一致。

### REV-03: sequential モードのプラン生成が非冪等（潜在・確定コード）

- 対象: `app/Services/GenerateProgramDayPlansService.php:110-124`（resolveDayTemplate）と `:72-88`（existing チェック） / migration `2026_07_17_100001`（routine_plans への index）
- 問題: `resolveDayTemplate` は「この version で過去に使われた全テンプレート」を除外して次の sequential DAY を返す。2 回目の呼び出しでは 1 回目に作った DAY が used に入るため**次の DAY が選ばれ**、`program_day_template_id` をキーにした existing チェックをすり抜けて**同じ日に 2 つ目のプランが作成**される。`/today` はロード毎に生成を呼ぶので、リロードするたび sequential ポインタが進む。現状はシード（`InstallElevenWeekProgramService`）が WeekdayFixed のみのため発火しないが、サービス・enum・factory は sequential を公称サポートしており時限バグ。
- 修正案: ①existing チェックを「その日にこの program version のプランが既にあるか」（template 不問）を先に見る形へ変更 ②DB に `unique(user_id, program_version_id, scheduled_on)` 相当の制約を追加（現状 plain index のみ）。
- 追加テスト: sequential モードで同一日 2 回生成 → プランが 1 件のまま・DAY ポインタが進まないこと。

### REV-04: `metric` ルートバインディングが user 未スコープ（確定）

- 対象: `routes/web.php:39-41` / migration `2026_07_16_100004`（metrics の unique を `key` → `(user_id, key)` に変更）
- 問題: バインダは `Metric::query()->where('key', $value)->firstOrFail()` のままグローバル検索。per-user metric が同一 key を持てるようになったため、`/records/{metric}` 系が**他ユーザーの Metric 定義を解決**しうる（どちらが返るかは不定）。
- 修正案: `where('key', $value)->where(fn ($q) => $q->whereNull('user_id')->orWhere('user_id', Auth::id()))->firstOrFail()`（共有メトリクス + 自分のメトリクスのみ）。user 固有を優先する orderBy を付けると衝突時も決定的になる。
- 追加テスト: 2 ユーザーが同一 key の metric を持つ状態で、各自のルートが自分の（または共有の）metric に解決されること。

### REV-05: ベースライン統計の破壊と stddev 未実装（確定）

- 対象: `app/Services/ComputeDailyResourceStatesService.php:78-100`
- 問題（2 点）:
  1. 同日チェックイン再送信時（`$existingToday` 真）、`mean_value` を**当日の生値そのもの**にリセットする（`$existingToday || $previousCount === 0 ? (float) $value : …`）。何十日分の running mean が 1 回の再送信で崩壊する。チェックインは `updateOrCreate` の通常フローなので容易に到達する。
  2. `stddev_value` を計算するコードがどこにも存在せず、z_load / readiness は常にフォールバック定数 `1.5`（`max(0.5, … ?? 1.5)`）で計算される。ADR-0011 の「データからの決定論」意図と乖離（→ S-03）。
- 修正案: ①再送信時は「当日分を差し替えた再計算」にする。最小実装は当日値を mean に入れない（`$existingToday` なら mean/sample_count を据え置き、EWMA のみ当日値で更新）。正攻法は日次履歴（daily_resource_states）から 28 日窓の mean/stddev を再集計。②stddev は同じ窓集計で算出し `stddev_value` に保存（Welford 法 or 単純再計算）。
- 追加テスト: 同日再送信後も mean_value が崩れないこと / 28 サンプル後に stddev_value が実データ由来になること。

## 仕様確認・提案（採否はオーナー判断）

- **S-01（#136 / MarginCalculator.php:314-325）当日収支の非対称**: 当日期日の「収入」は集計から除外（`dueOn > asOf`）、当日期日・期限超過の「支出」は算入。保守的（余裕を過小表示）で安全側だが、設計 doc に規定がない。→ 提案: この非対称を仕様として `yoyu-money-margin-design.md` §4 に明文化する（挙動は現状維持を推奨。変えるなら「confirmed の当日収入は算入」だが、入金時刻不確実性を考えると現状の方が思想に合う）。
- **S-02（横断）「今日」のタイムゾーン**: `users.timezone` と `UserTimezoneResolver` は #136 の Money 側にのみ導入され、self-management 側（`/today` プラン生成・checkin・meals）は `config/app.php` の UTC 基準のまま。JST ユーザーは **09:00 JST に日付が切り替わる**ため、早朝・深夜の運用で「今日」がずれる。→ 提案: `UserTimezoneResolver` を `TodayController` / `DailyCheckinController` / meals の日付決定に展開し、「日付の正はユーザー TZ」の原則を docs（today-ops.md）に明記。今回のレビューで最も採用価値が高い提案。
- **S-03（#137）stddev の扱い**: 現状は恒久的に固定 SD=1.5 で z_load / readiness を計算（REV-05-2）。→ 提案: ①実装するなら 28 日窓の再集計で stddev を持つ ②当面プレースホルダで行くなら ADR-0011 に「MVP は固定 SD、キャリブレーション後も近似」と明記して仕様と実装の乖離をなくす。
- **S-04（#138）前日コピーの重複時仕様**: 「既存記録がある日はコピー拒否」「マージ（追記）」「置換」のどれかを仕様として決める必要がある（REV-02 の前提）。→ 推奨: 拒否（メッセージ表示）が最も安全で実装も最小。
- **S-05（#138）チェックイン API の部分更新仕様**: REV-01 の修正と合わせ、`POST /daily-checkins` は「送信フィールドのみ更新（部分更新）」であることを records.md / today-ops.md に明記する。

## 良い点（維持すべき設計）

- **#136**: 金額は全経路 BIGINT minor + 文字列交換（`MoneyAmount` の bcmath、`^-?\d+$` バリデーション）で float 混入なし（唯一の float は概算明記の `LoanInterestEstimator`）。所有権は global scope（`BelongsToUser`）+ 明示 `withoutUserScope()->where('user_id')` + `assertOwned` の 3 層で、queue ジョブ内含め漏れなし。カード枠・借入可能額を資産に加算しない不変条件、simulation の fingerprint 409、退会時 purge 失敗で削除中断、rollback の void 方式はいずれも設計 doc どおり。
- **#137**: プログラム版の copy-on-write 改訂（ADR-0010 の記述以上に完全な deep copy）、シードに個人データを含めず `cleardawn:import-personal` で注入する構成、承認 A の決定論ルール→作戦カードの分離は仕様どおり。SM-D01〜D08 のバックログ記載はコードの実態と一致。
- **#138**: copy-previous-day のユーザースコープは正しく、テストで担保。intake は `GetDailyMealsQuery` 再利用で食事画面と計算が一貫。docs 更新は概ね実装と一致。
