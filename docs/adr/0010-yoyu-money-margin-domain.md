# ADR-0010: 「お金の余裕」をヨユウの意思決定支援ドメインとして実装する

- 状態: 承認済み
- 作成日: 2026-07-17
- 詳細設計: [ヨユウ「お金の余裕」設計・実装計画](../product/yoyu-money-margin-design.md)

## 文脈

既存の Finance 仕様は、収入・支出を一つの明細表へ保存して月次集計する簡易家計簿を想定している。
一方、今回必要なのは、残高・今後の入出金・カード請求・ローン返済・最低生活費・安全資金を重ね、
「この判断をすると今月と将来の余裕がどう変わるか」を比較する機能である。

簡易な収支明細だけでは、次を安全に表現できない。

- カード利用明細とカード請求・口座からのカード支払いの二重計上防止
- 支払予定と実取引の照合、部分支払い、延期
- リボ・分割・繰上返済の現在負担と将来負担の比較
- 判断前後の状態、想定効果、実際の効果の保存
- CSV の再取込、重複候補、取込取消

また、現行コードではヨユウが同一 Laravel アプリ・同一認証・同一 DB 上のプロダクトモジュールとして
`app/Domain/Yoyu`、`routes/yoyu.php`、`resources/js/pages/Yoyu` に実装されている。

## 決定

### 1. 配置

「お金の余裕」は別アプリや Clear Dawn 直下の Finance とせず、ヨユウ配下のサブドメインとして実装する。

- URL: `/yoyu/money/*`
- Backend: `App\Domain\Yoyu\Money\*`
- HTTP: `App\Http\Controllers\Yoyu\Money\*` / `App\Http\Requests\Yoyu\Money\*`
- Frontend: `resources/js/pages/Yoyu/Money/*`
- Layout: 既存 `OsShellLayout` とヨユウの `--os-yoyu*` トークンを再利用

### 2. MVP の会計モデル

MVP では複式簿記を採用しない。次の三層を分離した個人向け資金繰りモデルとする。

1. **残高事実**: 資金口座と残高スナップショット
2. **予定**: 収入・支払・カード請求・ローン返済を表す予定キャッシュフロー
3. **実績**: 手入力・CSV・将来の外部連携から入る実取引

予定と実績は照合テーブルで結び、同じ金額を予定と実績の双方から集計しない。

### 3. 金額

- DB は符号付き `BIGINT` の最小通貨単位（`*_minor`）を使い、浮動小数を使わない
- 金額本体は非負、入出金方向は `direction` で表す。口座残高だけは負値を許可する
- HTTP/TypeScript では精度事故を避けるため金額を文字列で受け渡す
- MVP の表示・入力通貨は JPY のみとするが、各主要レコードに `currency_code` を保持する
- 外貨換算・為替差損益・複数通貨合算は MVP 対象外

### 4. 余裕額

口座残高、カード利用可能枠、借入可能額を同一視しない。

- カード利用可能枠と借入可能額は資産・利用可能資金へ加算しない
- 「支払後残高」と「安全に使える余裕額」を別の値として表示する
- 安全に使える余裕額は、確度の高い収入、期限内の支出、未確定支出の予約、残り最低生活費、
  ユーザー設定の安全資金を決定的な計算で求める
- 不足は 0 に丸めず、符号付き見込み額と不足額を表示する

### 5. カードの二重計上防止

- 支出分析はカード利用明細を数える
- 資金繰りはカード請求予定を数える
- 口座からのカード支払いは `card_payment` として支出分析から除外する
- 確定請求がある期間は請求額を正とし、同期間の明細合計を資金繰りへ重ねない
- 未確定利用はカードスナップショットを優先し、明細合計と同時に加算しない

### 6. 定期支払いと月次履歴

- 定期条件はルールと生成済み予定を分離する
- `(recurring_rule_id, occurrence_on)` を一意にし、同じ予定を再生成しない
- 当月はライブ計算し、締めた月は改訂可能な月次スナップショットで保持する

### 7. 外部連携と AI

- MVP の入力は手入力と CSV のみ
- 銀行・カード・Moneytree LINK・Gmail 等の接続は MVP 対象外
- 金融明細を既存ヨユウチャット、朝ブリーフィング、キオクの AI 整理へ自動送信しない
- 将来の外部接続トークンは既存 connector パターンに合わせて暗号化し、Web リクエスト中に同期しない

## 採用しない案

### 別プロダクトとして切り出す

認証・設定・OS シェル・ヨユウの「行動するための余裕」という価値が重複し、ユーザーが時間・体調・お金を
別々の場所で判断することになるため採用しない。

### Clear Dawn の Finance を拡張する

Clear Dawn は目標・優先順位・実行の管理、ヨユウは実行可能性を作る役割である。資金繰りの調整や
「止める・延ばす・買う」の比較は後者に属するため採用しない。

### 複式簿記を最初から導入する

会計帳簿・税務・事業会計を目的としておらず、入力負荷と実装量が中心価値を上回るため採用しない。
自分口座間の振替、カード支払い、ローン元金は取引種別で収支集計から除外する。

## 理由

1. **既存整合**: ヨユウは既に `Domain/Yoyu`、専用ルート、OS シェル、ユーザー別スコープを持つ
2. **正確性**: 予定・実績・残高を分けることで、家計簿型一表モデルの二重計上を避けられる
3. **運用性**: CSV から始め、同じ実取引・照合境界へ将来の金融 API を追加できる

## 影響

- 旧 `finance_categories` / `finance_entries` ドラフトは実装せず、本 ADR と詳細設計で置き換える
- `resources/js/pages/Yoyu/Index.vue` へ機能を詰め込まず、Money ページ群を分割する
- ヨユウのサイドバーに「お金」を追加し、既存 Today タブの active 判定をパス対応へ直す必要がある
- ユーザータイムゾーンを共有設定として持つ
- 月次締め、CSV 原本保持、概算利息の許容範囲は本 ADR と詳細設計の採用値に従う
