# Skill: Test Design Review (No Swallowing Errors)

## 目的

- テストが「失敗を検知する」目的を失っていないかレビューし、設計を正道に戻す
- **実装/パッチは出さない**（提案のみ）
- テストの書き方の詳細基準は `test-quality-standards.mdc` を参照（本スキルは設計レビューに特化）

## 禁止（絶対）

- エラー握りつぶし（try/catch で無視、成功扱い、ダミー返却など）でテストを通す提案
- 実装詳細に依存するテスト（内部プロパティ前提、setAttribute 回数固定）へ寄せる提案
- `assertTrue(true)` を「テスト通過」の証拠にする提案

## 入力情報

- [対象テスト] / [対象メソッド/責務]
- [現状の問題]（壊れ方 / 握りつぶしになった経緯）
- [受入条件（Given/When/Then）]
- SEARCH_SCOPE

## 手順

1. **対象の実装コードを Read する**（テストだけ見て判断しない。テスト対象の責務を実コードで確認）
2. **同一ファイルの既存テストを最低2つ Read する**（アサーションパターンの把握。`test-quality-standards.mdc` の必須フロー）
3. **契約の明文化**: このテストが保証すべきビジネスルール（契約）を1〜3行で書く。書けない場合はテスト設計自体が誤り
4. **Unit vs Feature の切り分け**: 何を Unit（モック・ロジック分岐）で、何を Feature（実DB・統合）で担保するか決める
5. **Unit 最小案**: assert 対象とモック範囲を最小で設計する（`andReturn()` は実コードが戻り値を使う場合のみ）
6. **Feature シナリオ**: DB・日付・スコープ（他ユーザーの `user_id` データが返らないこと）等、実データで担保すべきシナリオを列挙する
7. **再発防止チェック項目**: 同種の壊れ方を防ぐレビュー観点を提示する

## 出力フォーマット（report-output-format.mdc 準拠で1コードブロックに収める）

1. 契約（保証すべきこと）
2. Unit vs Feature 切り分け
3. Unit 最小案（assert / モック範囲）
4. Feature テスト案（シナリオ）
5. 再発防止チェックリスト
