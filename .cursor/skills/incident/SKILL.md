# Skill: Incident Investigation → Minimal Fix

障害対応は「原因候補を3つ挙げて優先度を付ける → 最有力候補を検証で確定する → 最小修正」の順で進める。
バグ修正との違い: **ログ・スタックトレースが起点**であり、再現手順が不明なことが多い。
共通概念の定義は [../_shared/analysis-concepts.md](../_shared/analysis-concepts.md) を参照。

## 入力情報（ユーザーから受け取る / 不足していれば質問する）

- [タイトル] / [現象] / [期待] / [再現条件]
- [ログ]（スタックトレース / リクエスト / SQL / パラメータ）
- [補足]（直近変更 / PR / コミット — `git log` で直近変更を自分で確認してもよい）
- SEARCH_SCOPE（未指定なら着手前に対象範囲を提示して合意を取る）

## 手順

### Step 1: 原因候補の洗い出し
1. スタックトレース・ログから発生箇所（ファイル・行）を特定する
2. 直近の変更履歴（`git log` / PR）と発生タイミングの相関を確認する
3. 原因候補を **P1〜P3 の3つ** に絞り、それぞれ優先度の根拠を付ける

### Step 2: 検証
4. P1 を確定するための **最小の追加確認**（ログ・条件・SQL・パラメータ）を提示する
5. P1 確定のための検証手順（再現クエリ・データ条件）を具体化する

### Step 3: 修正と再発防止
6. P1 確定後の最小修正方針を提示する（確信度80%未満ならパッチ禁止）
7. 再発防止策を **最小で** 提案する（例外処理 / バリデーション / 監視 / テスト）
   - 過剰な防御コードを足さない（`architecture-layers.mdc`: FW を信頼する）
8. 影響範囲/確認観点、地雷チェック（10項目）を実施する

## 出力フォーマット（report-output-format.mdc 準拠で1コードブロックに収める）

1. 関連ファイル
2. 原因候補（P1〜P3 + 優先度の根拠）
3. 追加確認（最小）
4. 検証手順（P1確定）
5. 修正方針（最小差分）
6. パッチ案（確信度80%以上のみ）
7. テスト
8. 影響範囲/確認観点
9. 自己検証（確信度% / 根拠 / 不確実点 / 追加確認）
10. 地雷チェック結果（OK/NG）
