# 共通分析概念（全スキル共有リファレンス）

bugfix / incident / perf-review / review-only / spec の各スキルが参照する共通定義。
各スキルで「地雷チェック」「確信度」「Data Cardinality」等の語が出てきたら、この定義に従う。

---

## Data Cardinality（データ件数前提）

対象データが「0/1件・複数件・大量件」のどれかを、**推測ではなく Query / SQL / Eloquent のコードを根拠に**確定する。

判定手順:
1. 対象の Controller → Query / Service とデータ取得元まで遡る
2. クエリに `limit` / `first()` / ページネーションがあるか、`get()` / `all()` 全件取得か確認する
3. WHERE 条件（所有者・権限・日付範囲等）から本番での現実的な件数を見積もる

| 区分 | 目安 | 扱い |
|---|---|---|
| 0/1件 | `first()` / PK 指定 | ループ・Collection 操作を気にしなくてよい |
| 複数件 | 〜100件 | 一括処理可 |
| 大量件 | 1000件超になり得る | チャンク分割必須（`sql-memory-performance.mdc` 参照） |

## Filter Location（フィルタの実行場所）

絞り込みが **SQL（WHERE / JOIN ON）** で行われているか、**PHP（Collection の filter 等）** で行われているかを特定する。

- 原則: SQL 側で絞る。SQL フィルタ済みの処理を PHP フィルタに置き換える変更は危険信号
- PHP フィルタが許容されるのは、取得済みデータが少件数（〜100件）と確定している場合のみ

## Ops Delta（処理量の増減）

変更前後で以下がどう増減するかを具体的な数で示す。

- クエリ発行数（N+1 の発生・解消）
- ループ回数 / Collection 操作のパス回数
- メモリに載るレコード数

「増える」場合は最悪ケース（本番最大データ）で破綻しないことを説明する。

## 確信度（Confidence）

原因分析・修正提案の確実さを % で自己申告する。**80% 未満ならパッチ提案禁止**（調査項目の提示のみ）。

| 確信度 | 条件 |
|---|---|
| 90%〜 | 原因コード行を特定し、再現メカニズムを実コードで説明でき、修正で直る理屈が通る |
| 80%〜 | 原因コード行は特定したが、再現確認ができていない（環境・データ依存） |
| 〜79% | 原因が複数候補ある / コードでなくログ・推測ベース → **パッチ禁止・調査項目のみ** |

自己検証には必ず「確信度% / 根拠 / 不確実点 / 追加確認事項」の4点を含める。

## 地雷チェック（危険操作リスト）

修正提案・パッチ案を出す前に、以下すべてを OK/NG で確認し、結果を出力に含める。
NG が1つでもあれば、パッチではなく「提案のみ（理由/代替案/影響範囲）」に切り替える。

| # | チェック項目 | 根拠ルール |
|---|---|---|
| 1 | サイドバー/ヘッダー等の共通部品を変更していないか | `global-workflow.mdc` |
| 2 | 共通UIコンポーネント/パッケージの内部を直接変更していないか | `design-consistency.mdc` |
| 3 | データのアクセス範囲（`user_id`・Policy による権限）のスコープ漏れがないか | `sql-memory-performance.mdc` |
| 4 | 1000件超になり得るデータの全件取得・N+1 を生んでいないか | `sql-memory-performance.mdc` |
| 5 | テストが外部 HTTP を投げる可能性（Http::fake 漏れ）がないか | `http-safety.mdc` |
| 6 | ビルド・dev server 関連の禁止コマンドを実行していないか | `project-specific.mdc` |
| 7 | `dropColumn` と `addColumn` を同一クロージャに書いていないか | `sql-memory-performance.mdc` |
| 8 | 既存の CSS クラス・テンプレートクラスを削除していないか | `design-consistency.mdc` |
| 9 | エラー握りつぶし（try/catch 無視・成功扱い）でテストを通していないか | `test-quality-standards.mdc` |
| 10 | 差分最小か（整形・ついでリファクタ・命名変更の混入がないか） | `global-workflow.mdc` |

## SEARCH_SCOPE の規律

- ユーザー指定の SEARCH_SCOPE 内を主に探索する。スコープ外に原因があると判断した場合は、**勝手に広げず理由を添えて報告**する
- SEARCH_SCOPE が未指定の場合は、着手前に対象範囲を提示して認識を合わせる

## 出力の共通ルール

- 報告・レポート系の出力は `report-output-format.mdc` に従い、**1つの ```markdown コードブロックで囲む**
- コードブロック内にネストしたコードブロックを書かない（インラインコード・テーブルで代替）
