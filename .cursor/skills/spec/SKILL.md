# Skill: Spec Decision → Implementation Plan

## 目的

- 仕様・設計の論点を分解し、選択肢を比較して推奨案と実装方針を提示する
- **判断はユーザー（判断者）が行う**。AI は判断材料を揃える
- 共通概念の定義は [../_shared/analysis-concepts.md](../_shared/analysis-concepts.md) を参照

## 入力情報（ユーザーから受け取る / 不足していれば質問する）

- [テーマ] / [現行] / [問題]
- [制約]（現行互換 / 運用負荷 / 金銭トラブル防止など）
- [関連画面/URL] / [判断者] / [近い既存機能]
- SEARCH_SCOPE

## 手順

### Step 1: 現状把握
1. 近い既存機能を Read し、現行の構造・パターンを把握する
2. 関連する DB テーブル・Resource・画面のデータフローを追跡する

### Step 2: 論点分解と比較
3. 論点を **最大7つ** に分解する（多すぎる場合は主要論点に絞り、残りは補足）
4. 各論点に A/B 案とメリット・デメリットを付ける
5. 推奨案を選び、理由を3つ付ける（既存整合 / 性能 / 運用の観点）

### Step 3: 実装計画
6. 実装方針を具体化する（候補ファイル / DB 影響 / 共通部品の再利用）
7. 受入条件を Given/When/Then で書く
8. ユーザーが決めるべきことを **1行の質問** に絞って提示する
9. 影響範囲とテスト計画（Feature / Unit / Vitest の使い分け）を示す

## 注意

- 確信度80%未満の事項は断定しない（「〜のはず」ではなく「要確認」と明記する）
- 想定データ件数・パフォーマンス影響を実装方針に含める（`sql-memory-performance.mdc`）

## 出力フォーマット（report-output-format.mdc 準拠で1コードブロックに収める）

1. 論点（最大7）
2. 選択肢A/B（メリデメ）
3. 推奨案（理由3つ）
4. 実装方針（候補ファイル / DB影響 / 共通再利用）
5. 受入条件（Given/When/Then）
6. 決めるべき1行質問
7. 影響範囲/テスト計画
8. 自己検証（確信度% / 根拠 / 不確実点 / 追加確認）
